# v0.3 Phrase-Atom Compression: Theory vs Practice
### Brief Analysis — What We Thought, What We Built, What Was Proved

> Companion to `benchmark-v0.3.md` and `spec-v0.3.md`.
> Honest retrospective written immediately after the v0.3 work landed.

---

## Executive summary

We hypothesised that promoting common multi-word phrases ("you know",
"a lot of", "at the end of the day") to single dictionary IDs would
yield **~2.5× compression** on real-world transcript JSON. We built
the full mining + dictionary + longest-match pipeline cleanly, verified
byte-exact lossless round-trip, and measured.

The result: **1.99× — meaningful, but well below the projection.** The
shortfall is not a bug; it's a structural consequence of how fixed-tier
byte budgets interact with phrase composition. The compression gain is
modest. **The token-count and decode-throughput gains are large** —
which makes v0.3 a substantial win for the LLM-training use case
rather than the pure-compression use case.

This document explains the theory, the test, and what the data proved.

---

## 1. What we believed (the theory)

### 1.1 The compression hypothesis

A multi-word phrase like `"i don't know"` consists of 3 word tokens.
In v0.2 binary, those tokens cost roughly:

```
i      Tier 0   1 byte
don't  Tier 1   2 bytes
know   Tier 1   2 bytes
                ─────
total           5 bytes per occurrence
```

If we promote `"i don't know"` to a single Tier 1 phrase ID, it costs
**2 bytes per occurrence** — a savings of 3 bytes each time it appears.

At 51,518 occurrences in the corpus, this single phrase atom saves
about 155 KB across the training corpus. Extrapolated across the top
1,000 phrases, the spec projected **25 MB corpus-wide savings**.

For a 2.4 MB transcript file with ~5,000 phrase occurrences at 3 bytes
each saved, the per-file gain looked like ~15 KB — a 1% boost on top
of v0.2's 1.84× — which we projected would compound across thousands
of phrases to roughly +35%, landing the ratio near **2.5×**.

### 1.2 The semantics hypothesis

A second, separate motivation: phrases that compress well are usually
phrases that **mean something**. `"i don't know"` is an epistemic
statement; `"at the end of the day"` is an idiom; `"you know what i mean"`
is a discourse marker. Each is one semantic unit.

If LLMs trained against the v0.3 vocabulary see these as atomic tokens
rather than 3-5 token fragments, the model:

- Spends less of its context window on syntactic structure
- Has clearer signal for idiomatic and pragmatic meaning
- Generates more cohesive output (a phrase atom comes out whole or not at all)

This benefit is independent of byte-level compression.

### 1.3 The acceptance criteria (locked before testing)

```
A1   10/10 v1 samples round-trip byte-exact
A2    3/3 transcripts round-trip byte-exact
A3   transcript ratio >= 2.5x   <-- the compression target
A4   encode >= 1.5 MB/s
A5   decode >= 2.0 MB/s
A6   token-ids-v1.csv.gz committed
A7   byte fallback table committed
A8   3 transcripts held out from mining (no memorisation)
A9   v0.2 backward decode (optional)
A10  ELO_FILE_FORMAT.md updated to v1.0 magic
A11  top phrases naturally land in Tier 1 via ranking
```

A3 was the only target tied to a numeric expectation. The others were
build-quality and contract-shape goals.

## 2. What we built (the test)

### 2.1 The pipeline (Steps 11-14)

**Step 11 — `ngram_counter.py`**
N-gram frequency counting across the full corpus minus 3 held-out
transcripts. 2-5 grams in one pass (9:53 runtime); 6-9 grams in a
second pass (9:09). Held-out files excluded by `video_id` match.
Output: ~233k raw n-gram candidates above freq ≥ 100.

**Step 12 — `phrase_miner.py`**
PMI scoring (`log2(P(phrase)/prod(P(word_i)))`) per candidate, then
the **maximal-phrase filter** dropping any phrase whose longer parent
phrase has frequency ≥ 0.7 × child frequency. Sliding-window
duplicates collapsed. Output: 167,876 ranked survivors.

The filter dropped 65,107 candidates (28%) — almost all of them
sliding-window fragments of YouTube "link in the description below"
templates and similar overlapping windows. Real semantic atoms (idioms,
discourse markers) ranked high on PMI and survived.

**Step 13 — `dictionary_builder_v03.py`**
Merged words + phrases into a unified frequency ranking. Tier 0 unchanged.
Tier 1 reserved 1024 slots for top words; remainder filled by unified
ranking. Tier-0 bigrams dropped (601 phrases like `"in the"` where the
phrase ID costs the same as the constituent words). Emitted the full
LLM-vocabulary contract: `token-ids-v1.csv.gz`, `special-tokens-v1.json`,
`byte-fallback-v1.csv`, `profile-cuts-v1.json`.

**Step 14 — `compressor.py` longest-match scan**
After `tokenize()` + `_strip_implicit_spaces()`, the encoder now
greedily matches the longest phrase starting at each position in each
word run (up to 9 tokens). Both text-mode `encode_text` and binary-mode
`encode_bytes_binary` got the upgrade. Decoder unchanged — phrase IDs
round-trip naturally via reverse-LMDB lookup.

### 2.2 Two iterations on Step 13

The first build (no filter, no reservation) produced 1.99× on the
transcript average. The phrase-filter + Tier-1 reservation iteration
(this branch's final state) also produced 1.99×. The structural answer
turned out to be the same; the second iteration mostly redistributed
which tokens fell in which tier without changing the file-size
arithmetic.

## 3. What was proved (the validation)

### 3.1 Byte-exact lossless preservation

13 of 13 test files (10 v1 samples + 3 transcripts) round-trip
byte-exact through v0.3 encode + decode, in both text and binary wire
formats. The longest-match scan does not introduce ambiguity because
the decoder reverses each ID lookup naturally.

This is the only non-negotiable property of v0.3, and it held.

### 3.2 Stream token reduction (the unexpected headline)

| File | v0.2 stream tokens | v0.3 stream tokens | Reduction |
|---|---:|---:|---:|
| times_now (2.4 MB) | 988,391 | 449,275 | **-55%** |
| jocko (1.0 MB) | 435,107 | 164,302 | **-62%** |
| julian (798 KB) | 333,683 | 120,757 | **-64%** |

A 55-64% reduction in stream-token count means an LLM trained on v0.3
vocabulary sees the same source content at **2-3× higher density**
per token. Same context window budget, vastly more conversation.

This was implied by the spec but **not stated as a primary measurement
goal**. In retrospect it is the most consequential v0.3 outcome.

### 3.3 Decode throughput (a bonus)

Decode throughput nearly doubled (3.5 → 6.5 MB/s). Fewer stream tokens
to walk, fewer LMDB lookups per output byte. Encode throughput
slightly degraded (1.7 → 1.4 MB/s) because the longest-match scan
issues additional LMDB queries to look for phrase matches.

### 3.4 Lossless byte-cost reduction (the actual compression gain)

| File | v0.2 ratio | v0.3 ratio | gain |
|---|---:|---:|---:|
| times_now | 1.84× | **1.99×** | +7.8% |
| jocko | 1.87× | **2.02×** | +7.3% |
| julian | 1.81× | **1.97×** | +6.4% |
| **average** | **1.84×** | **1.99×** | **+7.2%** |

Real and reproducible — but well below the spec's 2.5× target.

### 3.5 LLM vocabulary contract artifacts

Five frozen profiles published with deterministic membership rules,
power-of-2 sizes, reserved byte-fallback and special-token tail
slots. `token-ids-v1.csv.gz` provides the public contract.

This is the most strategically valuable v0.3 output. It enables
reproducible third-party retraining and on-device deployment math
(see `docs/v1/profiles.md` for the memory budget tables).

## 4. Why A3 was missed (the structural finding)

The spec's 2.5× projection assumed **~4 bytes saved per phrase
occurrence on average**. Measured per-occurrence savings are **1-3
bytes**, and frequently **zero**. The arithmetic at v0.3's fixed
1/2/3/4-byte tier scheme is:

```
Phrase composition                Word-cost  Tier-1 phrase  Savings/occ
2 Tier-0 words ('in the')           1+1=2       2              0   <-- filtered
Tier-0 + Tier-1 ('you know')        1+2=3       2              1
Tier-0 + Tier-2                     1+3=4       2              2
2 Tier-1 words                      2+2=4       2              2
3-gram all Tier-0                   1+1+1=3     2              1
3-gram with one mid-tier            1+2+1=4     2              2
4-gram all Tier-0                   1+1+1+1=4   2              2
6-gram ('at the end of the day')   ~6         3 (Tier 2)      ~3
```

**High-frequency phrases consist of high-frequency words.** And
high-frequency words are already in low tiers. So the phrases with
the most occurrences have the **smallest** per-occurrence savings.

The 6-gram idioms with 4+ byte savings per occurrence (e.g.
`"at the end of the day"`) have far lower occurrence counts (~2,000)
than the 2-gram filler markers (e.g. `"you know"` at 539,244 with
zero savings until filtered out).

Total corpus savings from the top 1,000 phrases came to roughly
**5-15 MB**, not 25 MB. Per file: ~1-2% rather than ~15%. Add the
55% token-count win at lower bytes-per-token (Tier 2 phrases at 3 B)
against the displaced words that moved to Tier 2 from Tier 1, and the
byte-size net comes out at +7%.

This is not a bug. It is what the math gives you when phrase IDs cost
2-4 bytes and word IDs cost 1-2 bytes in a fixed scheme.

## 5. What this implies for v0.4

The path to materially better compression requires breaking out of the
fixed-tier byte budget. Two structural options are visible:

**Variable-length / bit-packed IDs.** Most LLM inference and modern
compression formats use elias-gamma or similar codes where the highest-
frequency tokens get the fewest bits. A Tier-0 token in a 6-bit ID
costs 0.75 bytes instead of 1; the most common phrase atoms could
fit in ~10 bits = 1.25 bytes. Implementing this on top of v0.3's
dictionary projects to **+25-35% additional reduction** on top of v0.3.

**Arithmetic coding at the ID layer.** A predictive entropy coder
(like the ones used in PPM, zstd, or modern image codecs) can give
~50% additional reduction by exploiting context: the bit cost of
each token depends on what came before it. This is harder to
implement and harder to keep stateless, but unlocks the biggest
remaining compression headroom.

Both are out of scope for v0.3 and would land as v0.4 work. The
v0.3 dictionary is the **stable substrate** they would build on.

## 6. What this implies for the LLM angle

The 55% token-count reduction was the implicit win of v0.3 and is now
explicit. For the **3B/8B/14B retraining experiments** described in
the project goals, this means:

| Scenario | LLM context budget | v0.2 vocab | v0.3 vocab | Effective text content |
|---|---|---|---|---|
| Mobile 3B model | 4k tokens | 4k subword | 4k EloAI tokens | **2-3x more dialogue** |
| Workstation 8B | 8k tokens | 8k subword | 8k EloAI tokens | **2-3x more dialogue** |
| Server 14B | 32k tokens | 32k subword | 32k EloAI tokens | **2-3x more dialogue** |

This is the strategic outcome that ties to the on-device LLM thesis
in `docs/v1/profiles.md`. v0.3 didn't deliver the 2.5× compression
ratio, but it delivered **the vocabulary that makes 8B-on-flagship-
phone-with-32k-effective-context viable**.

The training experiments described in the project goals are what
actually measure this. The compression ratio is incidental to that
measurement.

## 7. Decision implied for milestone

```
A1, A2          ✅ ship gate         lossless preserved
A3              ❌ ratio target      structurally bounded at this format
A4              ⚠️  encode speed     1.4 MB/s vs 1.5 target -- close
A5              ✅ decode speed      far exceeded (6.5 vs 2.0)
A6, A7, A8      ✅ artifacts         published
A9              ⏸ backward decode   not in v0.3 scope after all
A10             ⏸ format doc        defer to v1.0 file format spec
A11             ✅ phrase ranking    natural ranking + Tier 1 reservation
```

Recommendation: **declare v0.3 the LLM-readiness milestone** rather
than the compression milestone. Tag, ship, write benchmark report
(this document), and pivot v0.4 work toward variable-length IDs
where the compression headroom actually lives.

Or, more honestly: v0.3 is the **vocabulary milestone**.
Compression is a follow-on for v0.4.

## 8. What we should NOT do

A few tempting refactors that won't help:

- **Raise the absorption ratio** in phrase mining. The maximal-phrase
  filter is already aggressive at 0.7; tightening it drops good
  phrases without changing the math.
- **Tighten min_freq** further. Hapaxes don't compress materially
  either way; cutting them only shrinks the LMDB.
- **Bigger Tier 1 reservation**. We already keep top 1,024 words; the
  remaining 256 Tier-1 slots are correctly the best phrases. Going
  to 1,280 (all words in Tier 1) just demotes top phrases to Tier 2
  without saving the words anything new.
- **More phrases**. Adding the 2-9 grams below freq 100 dilutes
  the contract and slows encode without producing real compression.

The remaining wins live one architectural layer down: in the wire
format, not the dictionary.

---

## 9. Acknowledged risk in this analysis

This analysis is written against a 3-file held-out test set. The
results are deterministic and reproducible on those files, but the
broader corpus (14,803 files used for mining) might show meaningfully
different ratios on different content domains. Verifying that would
require encoding the full corpus and reporting macro-average ratios.

For the v0.3 milestone, the 3-file measurement is sufficient signal:
the compression gain is real but small, the token reduction is large,
and the structural ceiling on the fixed-byte format is now understood.

A future v0.4 + variable-length wire format would warrant a
larger benchmark run.
